# Развёртывание и оптимизация инференса Qwen3.5 на llama.cpp

## 1. Характеристики локального устройства

| Компонент | Значение |
|---|---|
| CPU | 11th Gen Intel(R) Core(TM) i7-11390H @ 3.40 GHz |
| Количество физических ядер | 4 |
| Количество логических потоков | 8 |
| Максимальная частота по данным Windows | 3418 MHz |
| DRAM | 16 GB |
| Модули памяти | 8 × 2 GB Micron Technology |
| Частота памяти | 4267 MHz |
| GPU | Intel(R) Iris(R) Xe Graphics |
| Выделенная память GPU по данным Windows | 1 GB |
| NVIDIA GPU | отсутствует |
| CUDA | недоступна |
| Основной режим запуска | CPU-only |

Так как на устройстве отсутствует дискретная NVIDIA GPU и недоступна CUDA, инференс выполнялся в режиме CPU-only. Интегрированная Intel Iris Xe Graphics не использовалась как основной ускоритель, поскольку объём выделенной памяти составляет около 1 GB, что недостаточно для стабильного GPU-инференса языковой модели.

## 2. Выбор модели и числовой точности

Для экспериментов была выбрана модель:

| Параметр | Значение |
|---|---|
| Семейство | Qwen3.5 |
| Модель | Qwen3.5-0.8B |
| Формат | GGUF |
| Квантование | Q4_K_M |
| Файл модели | `Qwen3.5-0.8B-Q4_K_M.gguf` |
| Размер файла модели | около 497 MB по выводу `llama.cpp` |
| Количество параметров | 752.39 M по выводу `llama.cpp` |

Выбор модели `Qwen3.5-0.8B` обусловлен ограничениями локального устройства. Более крупные модели семейства Qwen3.5 могут запускаться медленно в CPU-only режиме и сильнее нагружать оперативную память. Квантование `Q4_K_M` выбрано как компромисс между размером модели, скоростью инференса и качеством ответа. Для CPU-only инференса уменьшение размера весов особенно важно, так как этап декодирования токенов часто ограничен пропускной способностью памяти.

## 3. Развёртывание модели на llama.cpp

`llama.cpp` был собран локально из исходного кода с использованием Visual Studio Build Tools, CMake и NMake. После сборки в каталоге `build/bin` были получены необходимые исполняемые файлы:

```text
llama-cli.exe
llama-server.exe
llama-bench.exe
```

Тестовый запуск через `llama-cli` подтвердил, что модель успешно загружается и генерирует ответ.

Для дальнейших измерений использовался `llama-server`, запущенный как локальный OpenAI-compatible HTTP API сервер на `127.0.0.1:8080`.

Базовая команда запуска сервера:

```cmd
build\bin\llama-server.exe -m C:\qwen35_llamacpp_task\models\Qwen3.5-0.8B-Q4_K_M.gguf --host 127.0.0.1 --port 8080 -c 2048 -t 4 -b 512 -ub 512 --cache-ram 0 --chat-template-kwargs "{\"enable_thinking\":false}"
```

Проверка работоспособности сервера выполнялась через эндпоинт:

```cmd
curl.exe http://127.0.0.1:8080/health
```

Сервер вернул:

```json
{"status":"ok"}
```

## 4. Методика измерения TTFT и TPOT

Для измерений использовался фиксированный набор из 5 входных промптов, связанных с предметной областью проекта `AI-Adaptive-Training-Coach`. Каждый промпт запускался несколько раз. В baseline и оптимизационных экспериментах использовалось 3 запуска на каждый промпт, то есть 15 измерений на конфигурацию. В финальном замере использовалось 5 запусков на каждый промпт, то есть 25 измерений.

Используемые метрики:

| Метрика | Описание |
|---|---|
| TTFT | Time To First Token, время от отправки запроса до получения первого токена |
| TPOT | Time Per Output Token, среднее время генерации одного выходного токена после первого |
| Total time | Полное время обработки запроса |
| P95 | 95-й перцентиль, используемый для оценки редких более медленных запросов |

Для измерения TTFT использовался streaming-режим API, так как обычный non-streaming запрос возвращает ответ целиком и не позволяет точно измерить время появления первого токена.

Параметры генерации были зафиксированы:

| Параметр | Значение |
|---|---|
| `temperature` | 0 |
| `max_tokens` | 100 |
| Количество промптов | 5 |
| Количество запусков baseline | 3 на промпт |
| Количество запусков final | 5 на промпт |
| Prompt cache | отключён через `--cache-ram 0` |
| Thinking mode | отключался через `enable_thinking=false` и `/no_think` в промптах |

## 7. Анализ влияния параметров на узкие места

### 7.1. Параметр `--threads`

Изменение числа CPU-потоков оказало наибольшее влияние на результат.

| Конфигурация | Mean TTFT, с | Mean TPOT, с/токен | Вывод |
|---|---:|---:|---|
| `-t 2` | 0.387435 | 0.030747 | Недостаточно потоков, результат хуже baseline |
| `-t 4` | 0.275305 | 0.021434 | Базовая конфигурация |
| `-t 6` | 0.235190 | 0.019676 | Лучше baseline |
| `-t 8` | 0.230483 | 0.018703 | Лучший результат среди оптимизационных запусков |

Параметр `--threads` влияет на compute-bound часть инференса, прежде всего на обработку входного промпта. Также он влияет на декодирование, но декодирование в языковых моделях часто ограничено не только вычислениями, но и пропускной способностью памяти. На данном устройстве лучший результат был получен при `-t 8`, что соответствует числу логических потоков процессора Intel Core i7-11390H.

### 7.2. Параметры `--batch-size` и `--ubatch-size`

Конфигурация `-b 256 -ub 256` почти не изменила результат относительно baseline. Конфигурация `-b 1024 -ub 512` ухудшила и TTFT, и TPOT.

| Конфигурация | Mean TTFT, с | Mean TPOT, с/токен | Вывод |
|---|---:|---:|---|
| `-b 256 -ub 256` | 0.275755 | 0.021435 | Почти без изменений |
| `-b 512 -ub 512` | 0.275305 | 0.021434 | Baseline |
| `-b 1024 -ub 512` | 0.289521 | 0.022953 | Хуже baseline |

Эти параметры в основном влияют на prefill, то есть на compute-bound часть инференса. На данном CPU увеличение batch size до 1024 не дало выигрыша, вероятно, из-за роста накладных расходов и ухудшения эффективности работы с кэшами процессора.

### 7.3. Параметр `--ctx-size`

Проверялись контекстные окна `1024`, `2048` и `4096`.

| Конфигурация | Mean TTFT, с | Mean TPOT, с/токен | Вывод |
|---|---:|---:|---|
| `-c 1024` | 0.271502 | 0.021758 | TTFT немного лучше, TPOT немного хуже |
| `-c 2048` | 0.275305 | 0.021434 | Baseline |
| `-c 4096` | 0.265445 | 0.022357 | TTFT лучше, TPOT хуже |

Размер контекста влияет на memory capacity-bound и memory bandwidth-bound аспекты инференса, так как увеличение контекста увеличивает объём KV-cache. В данном эксперименте промпты были короткими, поэтому изменение `--ctx-size` не дало значимого ускорения. Более крупное контекстное окно может быть полезно для длинных запросов, но оно увеличивает расход памяти.

### 7.4. Параметр `--mlock`

Конфигурация с `--mlock` оказалась хуже baseline:

| Конфигурация | Mean TTFT, с | Mean TPOT, с/токен | Вывод |
|---|---:|---:|---|
| baseline | 0.275305 | 0.021434 | Без `--mlock` |
| `--mlock` | 0.284033 | 0.022199 | Хуже baseline |

`--mlock` может быть полезен в сценариях, где операционная система вытесняет данные модели в файл подкачки. В данном случае модель небольшая и полностью помещается в RAM, поэтому фиксация модели в памяти не дала преимущества.

## 9. Сравнение baseline и final

| Метрика | Baseline | Final | Изменение | Ускорение |
|---|---:|---:|---:|---:|
| Mean TTFT, с | 0.275305 | 0.229948 | лучше на 16.48% | 1.20× |
| Mean TPOT, с/токен | 0.021434 | 0.018153 | лучше на 15.31% | 1.18× |
| Mean total time, с | 2.397497 | 2.027313 | лучше на 15.44% | 1.18× |

В результате оптимизации средний TTFT уменьшился с 0.275305 с до 0.229948 с, а средний TPOT уменьшился с 0.021434 с/токен до 0.018153 с/токен. Основной вклад в улучшение внёс параметр `--threads`, так как переход с 4 на 8 потоков позволил эффективнее использовать логические потоки CPU.

## 10. Итоговый вывод

Модель `Qwen3.5-0.8B-Q4_K_M.gguf` была успешно развёрнута локально с использованием `llama.cpp` в режиме CPU-only. Базовая конфигурация показала средний TTFT 0.275305 с и средний TPOT 0.021434 с/токен. После серии оптимизационных экспериментов была выбрана конфигурация `-t 8 -b 512 -ub 512 -c 2048`, которая показала средний TTFT 0.229948 с и средний TPOT 0.018153 с/токен.

Наиболее эффективной оптимизацией оказалось увеличение числа потоков CPU до 8. Остальные параметры либо почти не изменили результат, либо ухудшили производительность на данном устройстве. Это объясняется тем, что модель имеет небольшой размер, полностью помещается в оперативной памяти, а инференс выполняется без дискретного GPU. Для данного локального устройства оптимальная стратегия состоит в использовании небольшой квантованной модели и максимального числа логических потоков CPU без чрезмерного увеличения batch size.
