# Проектирование набора инструментов для агента AI Adaptive Training Coach

## 1. Сценарий использования агента и среда, в которой он работает

### 1.1. Назначение агента
Агент работает как серверная AI-подсистема для персонализированных тренировочных рекомендаций. Он принимает структурированный JSON-запрос, анализирует профиль пользователя, историю тренировок и, при наличии, текущую тренировочную сессию, после чего возвращает либо стартовый план, либо адаптацию следующей тренировки, либо безопасный отказ.

### 1.2. Рабочая среда
Агент встроен в FastAPI-приложение и вызывается через endpoint `POST /coach`. На вход подаётся модель `CoachRequest`, содержащая:
- `user_profile`;
- `session_history`;
- `current_session`;
- `temperature`.

На выходе сервис должен вернуть `CoachResponse`.

В текущей реализации логика опирается на OpenAI-compatible API, structured output по схеме `CoachSGRResponse`, системный prompt с жёсткими правилами безопасности и локальный retriever по базе знаний `knowledge_base.json`.

### 1.3. Основные режимы работы
У агента есть два рабочих режима:
- `initial_plan` — когда история тренировок и текущая сессия отсутствуют;
- `adaptation` — когда есть история тренировок и/или текущая сессия.

### 1.4. Базовые правила принятия решений
В проекте уже зафиксирован приоритет правил:
1. медицинская безопасность;
2. ограничения пользователя;
3. признаки перегрузки;
4. прогрессивная перегрузка.

Если обнаружен медицинский риск, агент не должен продолжать обычную адаптацию и обязан вернуть отказ от тренировочной рекомендации.

### 1.5. Почему здесь нужны инструменты
Сейчас агент в основном реализован как один LLM-вызов с retrieval-блоком и последующей нормализацией ответа. Для следующего этапа проекта целесообразно сделать архитектуру инструментов, чтобы:
- вынести детерминированные вычисления из prompt в Python-функции;
- сократить число случаев, когда безопасность зависит только от текста модели;
- сделать reasoning и вызовы функций наблюдаемыми и тестируемыми;
- встроить Function Calling без ломки текущей схемы `CoachSGRResponse` / `CoachResponse`.

## 2. Предлагаемый набор инструментов

Ниже приведён минимально достаточный набор инструментов, который хорошо ложится на текущую архитектуру проекта и при этом реалистичен для последующей реализации.

| Инструмент | Тип | Назначение | Вход | Выход |
|---|---|---|---|---|
| `prepare_request_context` | Детерминированный вычислительный инструмент | Нормализует вход, определяет режим, обрезает историю до последних релевантных сессий, формирует единый контекст для агента | `user_profile`, `session_history`, `current_session` | `mode`, нормализованный профиль, сокращённая история, список ключевых фактов контекста |
| `retrieve_training_knowledge` | Knowledge / retrieval tool | Извлекает из локальной базы знаний упражнения, безопасные замены и подсказки по адаптации под конкретный запрос | Нормализованный контекст пользователя | Список релевантных документов: `title`, `category`, `tags`, `content`, `score` |
| `assess_safety_risk` | Guardrail tool | Выявляет медицинские риски и определяет, обязателен ли отказ от рекомендации | Нормализованный контекст, текущая сессия, ограничения, заметки пользователя | `medical_risk_detected`, `risk_signals`, `refusal_required`, `refuse_reason` |
| `assess_training_load` | Детерминированный аналитический инструмент | Выявляет признаки прогресса и перегрузки по RPE, выполнению подходов, сну и усталости | История, текущая сессия | `progress_detected`, `supporting_facts`, `recommended_progression`, `overload_detected`, `overload_signals`, `recommended_adjustment` |
| `apply_restriction_policy` | Policy tool | Проверяет конфликт решения с ограничениями пользователя и предлагает допустимые модификации или исключения упражнений | Профиль, ограничения, черновое решение, retrieval-контекст | `restrictions_present`, `limiting_factors`, `restriction_impact_summary`, допустимые замены/запреты |
| `build_training_recommendation` | Orchestration / synthesis tool | Собирает итоговое решение по следующей сессии на основе результатов всех предыдущих инструментов и политики приоритетов | Результаты `prepare_request_context`, `retrieve_training_knowledge`, `assess_safety_risk`, `assess_training_load`, `apply_restriction_policy` | Структура итоговой рекомендации: `decision`, `exercise_changes`, `reasoning`, `long_term_recommendation`, `safety_warnings`, `refused`, `refuse_reason` |
| `request_user_confirmation` | Interaction / confirmation tool | Формирует запрос на подтверждение, если входные данные противоречивы, неполны или рекомендуемое изменение слишком существенно для автоматического применения | Причина подтверждения, черновое решение, конфликтующие данные | `confirmation_required`, `question`, `options`, `blocked_action` |

## 3. Логика использования инструментов агентом

### 3.1. Базовая последовательность
1. Агент всегда начинает с `prepare_request_context`.
2. После нормализации контекста агент вызывает `retrieve_training_knowledge`.
3. Затем обязательно вызывается `assess_safety_risk`.
4. Если `refusal_required = true`, дальнейшее планирование блокируется, и агент сразу переходит к `build_training_recommendation` в режиме отказа.
5. Если медицинского риска нет, агент вызывает `assess_training_load`.
6. После этого агент вызывает `apply_restriction_policy`.
7. Затем вызывается `build_training_recommendation`.
8. Если есть неоднозначность или требуется подтверждение, вместо немедленной выдачи рекомендации агент использует `request_user_confirmation`.

### 3.2. Почему именно такой порядок
Этот порядок напрямую соответствует уже закреплённой в проекте иерархии правил: сначала безопасность, затем ограничения, затем признаки перегрузки и только потом прогрессия. Он также уменьшает риск того, что LLM сначала придумает опасное решение, а потом будет пытаться его «смягчить».

## 4. Риски и политики использования

### 4.1. Класс рисков: медицинский риск
**Примеры риска:** сообщения об острой боли, травме, боли во время движения, ухудшении состояния.

**Политика:**
- инструмент `assess_safety_risk` обязателен для каждого запроса в режиме `adaptation` и рекомендуется для `initial_plan`, если в ограничениях уже есть жалобы на состояние;
- при `medical_risk_detected = true` агент не имеет права продолжать подбор нагрузки;
- агент обязан вернуть отказ, а не «осторожную тренировочную рекомендацию»;
- список `exercise_changes` при отказе должен быть пустым.

**Human-in-the-Loop:**
- в MVP не используется подтверждение «продолжить несмотря на риск»;
- вместо этого запрос блокируется и переводится в безопасный отказ;
- при развитии продукта возможна эскалация к человеку-коучу или врачу, но не автоматическое продолжение.

### 4.2. Класс рисков: конфликт с ограничениями пользователя
**Примеры риска:** рекомендации, противоречащие ограничениям по колену, пояснице, запрещённым упражнениям или отсутствующему оборудованию.

**Политика:**
- инструмент `apply_restriction_policy` обязателен для всех запросов;
- ограничения пользователя имеют приоритет над прогрессией и над retrieval-подсказками;
- если упражнение конфликтует с ограничением, оно должно быть либо исключено, либо заменено безопасной альтернативой;
- retrieval не может отменить пользовательское ограничение.

**Human-in-the-Loop:**
- не требуется, если система может безопасно исключить или заменить упражнение;
- требуется, если пользовательские ограничения сформулированы противоречиво или слишком расплывчато, а безопасная замена неочевидна.

### 4.3. Класс рисков: некорректная адаптация нагрузки
**Примеры риска:** повышение нагрузки при `RPE > 9`, недосыпе, высокой усталости или недовыполнении подходов.

**Политика:**
- `assess_training_load` должен вычислять признаки прогресса и перегрузки детерминированно;
- при признаках перегрузки агент может только снижать интенсивность или объём;
- увеличение нагрузки допускается только если нет медицинского риска, нет конфликта с ограничениями и есть признаки прогресса.

**Human-in-the-Loop:**
- не требуется для обычной коррекции в пределах заранее определённых правил;
- требуется, если агент хочет предложить слишком резкое изменение, выходящее за допустимую автоматическую политику.

### 4.4. Класс рисков: некачественный retrieval
**Примеры риска:** irrelevant knowledge, советы общего характера, конфликт retrieval-контента с фактическими входными данными.

**Политика:**
- `retrieve_training_knowledge` используется только как вспомогательный источник;
- фактические данные пользователя и safety-правила всегда имеют более высокий приоритет;
- retrieval нельзя использовать как основание для отмены отказа по медицинскому риску;
- при низкой релевантности retrieval может быть проигнорирован.

**Human-in-the-Loop:**
- не требуется.

### 4.5. Класс рисков: неоднозначный или неполный вход
**Примеры риска:** отсутствует состав упражнений в текущей сессии, противоречие между ограничениями и заметками, некорректные числа, неясно — пользователь хочет стартовый план или адаптацию.

**Политика:**
- `prepare_request_context` должен выявлять неполноту и противоречия;
- если запрос не позволяет безопасно принять решение, агент не должен додумывать скрытые факты;
- в таких случаях запускается `request_user_confirmation`.

**Human-in-the-Loop:**
- требуется в форме подтверждения/уточнения пользователем;
- без подтверждения агент не должен выполнять заблокированное действие.

## 5. Где Human-in-the-Loop действительно нужен

Для текущего проекта Human-in-the-Loop нужен не везде, а только в узких местах.

### HITL не требуется
- стартовый план при корректном и полном профиле;
- обычная адаптация при полном наборе данных;
- автоматический безопасный отказ при медицинском риске;
- автоматическое снижение объёма/интенсивности при прозрачных признаках перегрузки.

### HITL требуется
- вход противоречив или недостаточен для безопасного решения;
- агент предлагает крупное изменение, выходящее за заранее заданную безопасную политику;
- ограничения пользователя описаны слишком расплывчато, и безопасная замена неочевидна.

## 6. Рекомендация по реализации в следующем задании

Для следующего этапа проекта инструменты стоит реализовывать в виде отдельных Python-функций с Pydantic-моделями входа/выхода. Затем их нужно зарегистрировать как `tools` в OpenAI-compatible API и заставить агента использовать их в жёстко заданной последовательности:

- сначала контекст;
- затем retrieval;
- затем safety;
- затем анализ нагрузки;
- затем применение ограничений;
- затем итоговое решение;
- при необходимости — подтверждение.

Такая схема остаётся совместимой с текущим FastAPI API, с уже существующей SGR-схемой и с тестовыми сценариями `initial_plan`, `progress`, `overload` и `medical risk`.

## 7. Итог

Для AI Adaptive Training Coach оптимален компактный набор из семи инструментов: подготовка контекста, retrieval, safety-проверка, анализ нагрузки, применение ограничений, сборка рекомендации и запрос подтверждения. Этот набор соответствует текущей декомпозиции системы и позволяет перенести критически важную логику из prompt-only подхода в наблюдаемую и тестируемую инструментальную архитектуру.
